## Clarifications

### Session 2026-07-23

- Q: 会话结束后的「打卡」如何触发？ → A: 结束后展示摘要；用户可点「打卡」才产生打卡载荷（可跳过）
- Q: Tabata「N 组」如何计段？ → A: 每组 = 训练 + 休息；共 N 组，含最后一组休息；全部结束后才完成
- Q: 训练计划唤起并带入参数后是否自动开始？ → A: 唤起后预填参数，等待用户确认「开始」
- Q: 自定义时长/组数的合法上限？ → A: 单段 1–3600 秒；Tabata 组数 1–50
- Q: 组间休息结束后下一步默认给什么？ → A: 摘要页提供「再用同时长」快捷重开（一键开始同配置）

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 组间休息倒计时 (Priority: P1)

训练者在组与组之间需要固定休息。打开计时器，选择「组间休息」，设定休息秒数后开始；屏幕以大字体显示剩余时间；时间到时震动并播放提示音，明确告知休息结束。

**Why this priority**: 组间休息是健身房最常用场景，单独交付即可形成可用 MVP。

**Independent Test**: 仅实现休息倒计时：设定时长 → 开始 → 倒计至 0 → 震动+提示音；可在前台完成全流程验收。

**Acceptance Scenarios**:

1. **Given** 用户进入计时器并选择组间休息，**When** 设定自定义时长并开始，**Then** 大字体显示剩余时间并按秒递减至结束
2. **Given** 倒计时进行中，**When** 到达 0，**Then** 触发震动与提示音，并给出可感知的「休息结束」反馈
3. **Given** 倒计时进行中，**When** 用户暂停再继续或取消，**Then** 行为符合预期（暂停冻结剩余时间；取消结束会话且可重新配置）
4. **Given** 休息倒计时已正常结束并进入摘要，**When** 用户点「再用同时长」，**Then** 以相同休息时长立即开始新一轮倒计时（无需重新填写）

---

### User Story 2 - Tabata 交替循环 (Priority: P1)

训练者进行高强度间歇：设定「训练时长」「休息时长」与「循环组数」N，开始后按「训练 → 休息」自动交替；每一组均包含训练段与休息段（含最后一组休息），共推进 `2N` 段，全部完成后结束并提醒。

**Why this priority**: 与休息倒计时同属宪章定义的三种核心模式之一，是间歇训练主路径。

**Independent Test**: 用短时长（如 3 秒训练 / 2 秒休息 / 2 组）跑完：应出现 2 次训练 + 2 次休息，第 4 段结束后进入完成态。

**Acceptance Scenarios**:

1. **Given** 用户选择 Tabata 并填入训练时长、休息时长、循环组数 N，**When** 开始，**Then** 界面标明当前阶段（训练/休息）、当前组次（1…N）与剩余时间
2. **Given** 一轮训练段结束，**When** 进入该组休息段，**Then** 触发切换提醒（震动+提示音），并开始休息倒计时
3. **Given** 第 N 组休息段结束，**When** 全部 `2N` 段完成，**Then** 给出结束提醒，会话进入完成态（最后一组仍含休息，不跳过）

---

### User Story 3 - 正计时记录训练时长 (Priority: P2)

训练者自由训练，需要知道已练多久。选择正计时并开始，大字体显示已用时间；可暂停/继续/结束；结束时可得到本次时长摘要。

**Why this priority**: 使用频率略低于固定间歇，但同属三种必备模式，可在 P1 模式稳定后交付。

**Independent Test**: 开始正计时 → 等待数秒 → 暂停 → 继续 → 结束，核对显示时长与真实经过时间一致（允许秒级误差）。

**Acceptance Scenarios**:

1. **Given** 用户选择正计时并开始，**When** 时间流逝，**Then** 大字体显示已用时间并递增
2. **Given** 正计时进行中，**When** 用户暂停再继续，**Then** 已用时间在暂停期间不增加，继续后从暂停点累加
3. **Given** 用户结束正计时，**When** 查看摘要，**Then** 能看到本次累计时长

---

### User Story 4 - 后台与锁屏下计时不中断 (Priority: P1)

训练中用户常熄屏或切到其他应用。进行中的计时会话在回到前台后，显示的剩余/已用时间须与真实墙钟时间一致；到点提醒仍应尽量准时触发（在微信小程序平台能力范围内）。

**Why this priority**: 无此后台能力，计时器在真实训练场景不可用。

**Independent Test**: 开始短时休息倒计时 → 立即切后台或锁屏 → 到点后回前台，核对时间已走完且提醒已触发或回前台瞬间补触达。

**Acceptance Scenarios**:

1. **Given** 倒计时进行中，**When** 用户将小程序切至后台或锁屏一段时间再返回，**Then** 显示时间按真实经过时间校正，无「暂停在切走那一刻」的错误
2. **Given** 计时在后台期间到达结束点，**When** 用户回到小程序（或平台允许的后台提醒时机），**Then** 结束提醒已发生或立即补发，用户能确认该段已结束

---

### User Story 5 - 独立入口与计划唤起、结束打卡预留 (Priority: P2)

用户可从训练模块独立打开计时器三种模式；训练计划（后续模块）也可一键唤起并带入预设（如休息秒数或 Tabata 参数）。一次计时结束后，系统可发出「打卡」意图：携带预定义的打卡数据结构，本功能不负责持久化存储，仅预留对接点便于后续联调。

**Why this priority**: 独立入口保证本功能可单独上线；唤起与打卡是集成边界，结构先定、存储后接。

**Independent Test**: 从训练入口完成一次完整会话；用模拟「唤起参数」打开计时器并看到预填配置；结束时观察到打卡载荷字段齐全（无需真实写入云库）。

**Acceptance Scenarios**:

1. **Given** 用户在训练模块，**When** 打开训练计时器，**Then** 可选择三种模式之一并开始，无需经过训练计划
2. **Given** 外部以约定方式唤起计时器并附带模式与时长参数，**When** 页面打开，**Then** 对应模式与参数已预填就绪，会话处于未开始的确认态；用户可微调后点「开始」，MUST NOT 因唤起而自动开跑
3. **Given** 一次计时会话正常结束，**When** 用户查看摘要并可选择「打卡」或跳过，**Then** 仅在用户主动点「打卡」时产生符合预定义结构的打卡载荷；跳过则不产生打卡意图；本版本不要求写入远端/本地持久库

---

### Edge Cases

- 自定义时长为 0、负数或超过上限（单段 > 3600 秒）时，拒绝开始并给出中文提示
- Tabata 循环组数为 0、非整数或超过 50 时，拒绝开始并提示
- 用户快速连点开始/暂停时，不产生重复会话或错乱阶段
- 提示音无法播放（静音、系统限制）时，震动仍须尽量触发；两者皆不可用时，界面须有明确结束态文案
- 切后台时间很长导致计时早已结束：回前台须直接进入完成态并补提醒，而不是从切走时继续倒
- 唤起参数缺失或非法时，回退到该模式的安全默认值并允许用户修改
- 小屏设备上主时间数字仍须一眼可读；主操作按钮适合单手点击

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 系统 MUST 提供训练计时器，支持三种模式：组间休息倒计时、Tabata、正计时
- **FR-002**: 组间休息模式 MUST 允许用户自定义休息时长；时间到 MUST 触发震动与提示音提醒
- **FR-003**: Tabata 模式 MUST 支持自定义训练时长、休息时长与循环组数 N；MUST 按「每组 = 训练段 + 休息段」推进，共 `2N` 段（含最后一组休息）；MUST 在段切换与全部完成时提醒
- **FR-004**: 正计时模式 MUST 从零累加显示已用训练时长，并支持暂停、继续与结束
- **FR-005**: 所有模式进行中 MUST 以简洁大字体展示当前关键时间；界面操作步骤少，适合训练中单手操作
- **FR-006**: 进行中会话 MUST 支持暂停、继续与取消/重置（正计时结束即得到时长摘要）
- **FR-007**: 计时 MUST 以真实经过时间为准：小程序进入后台或锁屏后，回前台显示与到点判定 MUST 与墙钟一致（在微信平台能力范围内尽量保证到点提醒）
- **FR-008**: 阶段切换（Tabata）与会话结束 MUST 触发震动；MUST 尝试播放提示音；若声音不可用，MUST 仍保留可视完成反馈
- **FR-009**: 用户 MUST 能从训练模块独立进入计时器并使用全部三种模式
- **FR-010**: 系统 MUST 提供可被训练计划（或其他训练页）唤起的入口约定：可传入模式与相关时长/组数预设；唤起后 MUST 预填参数并等待用户确认「开始」，MUST NOT 仅因唤起而自动开始计时
- **FR-011**: 会话结束时 MUST 展示摘要，并提供可选的「打卡」操作；仅当用户主动确认打卡时 MUST 产生符合预定义结构的打卡载荷；用户可跳过且 MUST NOT 因此被强制产生打卡意图；本功能 MUST NOT 依赖尚未建成的持久化存储即可验收结构正确性
- **FR-012**: 非法或越界的时长/组数配置 MUST 阻止开始并给出清晰中文提示；合法范围 MUST 为：单段时长 1–3600 秒（含）；Tabata 循环组数 1–50（含）
- **FR-013**: 计时行为（模式、阶段、剩余/已用时间、完成与提醒事件）MUST 以稳定、可复用的契约供独立页与唤起方共用，避免多处各自实现导致规则不一致；纯计时规则与设备提醒能力 MUST 可分开演进与验收
- **FR-014**: 面向用户界面 MUST 遵循品牌视觉约定（`docs/brand`），MUST NOT 另起主色板或引入重型 UI 组件库
- **FR-015**: 计时器相关文案 MUST NOT 包含医疗/治疗承诺；须避免鼓励超负荷或不安全训练；页面 MUST 提供简短使用提示或免责声明（仅供辅助计时，量力而行）
- **FR-016**: 组间休息模式在会话正常结束后，摘要 MUST 提供「再用同时长」快捷操作：一键以相同配置开始新一轮倒计时；MUST NOT 在结束后自动连续开跑（除非用户主动点快捷重开）

### Key Entities

- **TimerSession（计时会话）**：模式、状态（空闲/进行中/暂停/已完成/已取消）、开始墙钟、累计有效时长、当前阶段剩余或已用显示值
- **RestCountdownConfig**：休息时长（秒）
- **TabataConfig**：训练时长、休息时长、循环组数 N；当前组次与当前阶段（训练/休息）；完成条件为 `2N` 段全部结束（含末组休息）
- **StopwatchConfig**：无目标时长；记录累计已用秒数
- **TimerAlert**：提醒类型（阶段切换 / 结束）、是否震动、是否尝试播音
- **WorkoutCheckInPayload（打卡载荷）**：预定义字段，如模式、配置快照、实际用时、完成时间、来源（独立/计划唤起）；不含本版本强制持久化
- **TimerLaunchParams（唤起参数）**：模式与可选预设字段，供训练计划等调用方传入

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 三种模式均可在不依赖训练计划的情况下独立完成一次完整会话（配置 → 开始 → 结束提醒）
- **SC-002**: 在受控测试中，前台计时与墙钟误差在正常使用下不超过 1 秒（按秒显示场景）
- **SC-003**: 切后台或锁屏不少于 10 秒后返回，显示时间与真实经过时间一致（误差 ≤ 1 秒）；短时倒计时在后台到期时，用户能感知到结束提醒或回前台立即看到完成态
- **SC-004**: Tabata 短参测试（如 3s/2s/2 组）须完成 4 段（训练/休息/训练/休息）后结束；阶段切换与结束提醒次数符合配置，无跳段、无跳过末组休息
- **SC-005**: 新手受试在无讲解下，于 30 秒内能开始任一种模式计时（主路径操作步骤少、大字体可读）
- **SC-006**: 用户在摘要页点「打卡」后，载荷必填字段齐全、类型符合预定义结构；选择跳过时不产生打卡意图；无需真实云存储即可验收
- **SC-007**: 使用非法时长/组数（含 0、负数、单段 >3600 秒、组数 >50）时，100% 被拦截并显示中文提示，不进入错误运行态
- **SC-008**: 界面主色与品牌约定一致，无第二套主色板或重型组件库痕迹
- **SC-009**: 组间休息正常结束后，用户可在摘要页一键「再用同时长」开始下一轮；未点击时不会自动连续倒计时

## Assumptions

- 默认预设（用户可改）：组间休息 90 秒；Tabata 训练 20 秒 / 休息 10 秒 / 8 组（8×(训练+休息)）；正计时从 0 起
- Tabata 组定义：每组含训练与休息；最后一组休息不跳过
- 配置合法范围：单段时长 1–3600 秒；Tabata 组数 1–50
- 组间休息结束后摘要提供「再用同时长」快捷重开；不自动连环开跑
- 时长单位为秒；界面可显示为 `mm:ss` 或纯秒，以大字体可读为准
- 「训练计划一键唤起」在本功能交付时以约定参数与导航入口验收；完整计划库页面可尚未上线；唤起仅预填，不自动开始
- 打卡仅在用户于摘要页主动确认时产生载荷；跳过则无打卡意图；本地/云端持久化、列表展示归后续「训练打卡记录」能力
- 震动与提示音受设备与系统静音影响；无法保证所有机型后台必达系统级通知，但 MUST 用墙钟校准保证时间正确，并在能力范围内最大化提醒可达
- 纯计时状态推进与校验逻辑适合沉淀为可单测的共享模块；震动、音频、前后台生命周期适配留在小程序端
- 本功能不包含社交分享、多用户同步或 Web 端计时器复刻
- 品牌与大字体布局遵循 `docs/brand` 与现有小程序视觉气质，不引入新的主色体系

## Constitution Alignment _(mandatory)_

### In Scope / Out of Scope

- **In scope**: 微信小程序训练模块内的训练计时器（三种模式、提醒、后台墙钟校准、独立入口、计划唤起约定、打卡载荷预留）；TypeScript 类型化可复用计时核心
- **Out of scope**: 训练计划库正文、动作/器材库、打卡持久化与「我的」记录页、Web 计时器、社区、自建后端、重型 UI 库、暗黑模式专项

### Stack & Shared Logic

- **Language**: TypeScript（计时器状态、模式配置、回调/事件参数 MUST 完整类型定义；无充分理由不新增纯 JS）
- **Target package(s)**: `packages/miniprogram`（页面与平台适配）；可复用纯计时逻辑 MUST 进入 `packages/shared` 并 TDD
- **Shared extraction**: 模式配置校验、会话状态迁移、基于墙钟的剩余/已用时间推算、打卡载荷造型；MUST NOT 在 shared 中调用小程序 API
- **Backend usage**: 无；本功能不写云库。打卡存储后续对接 CloudBase 时另开需求

- [x] No medical/treatment claims; disclaimer required on calculators/guidance（计时辅助 + 量力而行提示）
- [x] Privacy: minimum necessary data only; privacy policy implications noted（本版不持久化打卡）
- [x] No extreme diet / overload training recommendations
- [x] Platform rules: WeChat category/content alignment (if miniprogram)
- [x] Brand: UI MUST consume docs/brand; calculators follow visual contract; no alternate primary palette / heavy UI kits
