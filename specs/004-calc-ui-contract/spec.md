# Feature Specification: 小程序计算页视觉契约（对齐品牌）

**Feature Branch**: `004-calc-ui-contract`

**Created**: 2026-07-23

**Status**: Draft

**Input**: User description: "功能名称：小程序计算页视觉契约（对齐品牌）。所属模块：packages/miniprogram。为 PeakMeet 微信小程序建立可复用的轻量视觉与交互契约，强制对齐 docs/brand/README.md 品牌色与 Logo，消除计算页样式开盲盒。品牌色 Ink/Orange/Snow；Logo 使用小程序 brand 资产；共享样式入口；存量工具箱与热量缺口页迁移；业务逻辑不变。"

## Clarifications

### Session 2026-07-23

- Q: 主操作按钮（「计算」）用什么填色？ → A: 主按钮 Ink 实心 + 浅色字；Orange 仅用于选中态、小点缀或主数字点睛
- Q: 本功能交付完成时存量迁移要做到哪一步？ → A: 本功能内完成清单全部对齐（工具箱五页 + 热量缺口）后才算交付完成
- Q: 饮食 Tab 首页与工具箱聚合页要不要纳入本功能？ → A: 纳入：饮食 Tab 入口 + 工具箱聚合页与计算页一并对齐品牌契约
- Q: 结果区「主数字」如何用色？ → A: 主数字 Ink；单位次级灰；可选极小 Orange 点缀（非整数字着色）
- Q: 性别/模式等「选中态」如何表现？ → A: 选中：Orange 描边 + 浅底 + Ink 字；未选中：中性浅灰底

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 建立可复用的品牌视觉契约 (Priority: P1)

产品与设计需要一份可执行的计算页视觉约定，使后续页面不再各自选色。交付后，任意计算页应使用同一套品牌色（墨色 Ink、强调橙 Orange、雪色 Snow）与清晰的信息层级：主结果数字最突出，说明与免责声明依次弱化但仍可见。

**Why this priority**: 没有统一契约，存量与增量页面会继续发散，品牌识别与新手可读性无法保证。

**Independent Test**: 仅凭契约说明与样板页，评审者能指出主色仅来自三品牌色、主数字层级正确、免责声明可见且不压过主结果。

**Acceptance Scenarios**:

1. **Given** 视觉契约已生效，**When** 查看任一已对齐的计算页，**Then** 页面主色仅来自 Ink / Orange / Snow（及由其衍生的深浅/透明度），无第二套主色（如紫渐变、奶油陶土风等）
2. **Given** 用户完成一次合法计算并看到结果，**When** 扫视结果区，**Then** 能在不阅读长文的情况下先识别「最重要的数字」（主数字为 Ink，单位为次级灰；若有 Orange 仅极小点缀，非整串数字为橙）
3. **Given** 结果已展示，**When** 查看页面底部区域，**Then** 免责声明可见，且视觉权重低于主结果
4. **Given** 页面使用强调色，**When** 对比主按钮/选中态与大面积背景，**Then** 主按钮为 Ink 实心；Orange 仅作选中态或点睛，未大面积铺满造成刺眼或喧宾夺主

---

### User Story 2 - 共享样式入口供计算页统一引用 (Priority: P1)

开发需要单一共享样式来源，避免每页复制互斥的私货样式。计算页统一引用该入口后，表单字段、主按钮、选择态、错误提示、结果区与免责声明观感一致。

**Why this priority**: 契约若不落地为可引用的共享样式，无法验收「统一」也无法约束增量页面。

**Independent Test**: 打开至少两个已对齐计算页，对比主按钮、表单字段、结果主数字与免责声明层级，观感一致且均引用同一共享样式来源。

**Acceptance Scenarios**:

1. **Given** 共享样式入口已提供，**When** 检查已对齐的计算页，**Then** 均通过该入口消费品牌 token，而非各页单独定义互斥主色
2. **Given** 用户在工具箱样板计算页与热量缺口页之间切换，**When** 对比主操作按钮与表单控件，**Then** 主按钮均为 Ink 实心；选中态均为 Orange 描边 + 浅底 + Ink 字、未选中为中性浅灰底；错误提示位置符合同一套约定
3. **Given** 新人要做新的计算页，**When** 阅读本功能附带的引用说明，**Then** 能知道如何引用品牌 token 与共享样式，而无需重新发明色板

---

### User Story 3 - 存量饮食计算页迁移到契约 (Priority: P1)

已上线的身体指数工具箱（聚合页 + BMI、基础代谢、体脂估算、腰臀比、1RM）、热量缺口计算页，以及饮食 Tab 入口页，须迁移到共享视觉契约。允许分批改页，但本功能交付前必须全部对齐；迁移不得改变既有计算与交互行为。

**Why this priority**: 用户从饮食 Tab 进入到各计算页是完整路径；只改计算页、入口仍旧皮，品牌体验仍不连贯。

**Independent Test**: 按迁移清单逐页核对（含饮食 Tab 与工具箱聚合页）：引用同一 token/共享样式；入口按钮与计算页主按钮同为 Ink 实心；原有导航与计算行为仍可用。

**Acceptance Scenarios**:

1. **Given** 迁移清单列出饮食 Tab 入口、工具箱聚合页、工具箱五计算页与热量缺口页，**When** 本功能交付完成，**Then** 清单内每一页均已对齐；不得以「仅计算页」宣称本功能完成
2. **Given** 某计算页已迁移，**When** 用户按原路径完成一次合法计算，**Then** 计算结果与校验行为与迁移前一致（仅视觉对齐，业务逻辑不变）
3. **Given** 实现过程中某页尚未改完，**When** 查看进度，**Then** 清单中仍可区分「已对齐 / 待对齐」；全部页对齐前不得关闭本功能的存量迁移验收
4. **Given** 用户打开饮食 Tab 或工具箱聚合页，**When** 查看入口与品牌识别，**Then** 使用约定品牌资产（icon 或 lockup，按需）；背景/文字/主入口按钮符合 Ink–Orange–Snow 契约，且不得用无关装饰淹没品牌

---

### User Story 4 - 增量计算页默认遵循契约 (Priority: P2)

此后新建的计算类页面默认遵循本视觉契约与品牌 token，产品验收时可直接按契约检查，无需再为每页单独发明主色。

**Why this priority**: 防止迁移完成后再次发散；可在契约与共享入口就绪后独立验收「规范已写清」。

**Independent Test**: 抽查本功能文档中的「新页面如何引用」说明，确认包含必须遵循品牌三色、共享样式入口与信息层级规则。

**Acceptance Scenarios**:

1. **Given** 计划新增一个计算页，**When** 查阅本功能规格/快速说明，**Then** 明确要求遵循本契约，且不得另起主色板
2. **Given** 增量页已按契约实现，**When** 与已迁移样板页对比，**Then** 主色与信息层级一致

---

### Edge Cases

- 仅使用三品牌色的透明度/灰阶衍生时，仍视为合规；引入与品牌无关的第四主色视为不合规
- Orange 用于风险提示或选中描边时，须可辨识但不得做成大面积恐慌夸张橙底而压过主结果
- 浅色容器（近白）可用于表单/结果块，但页面整体背景须落在 Snow 体系，避免「另一套皮肤」
- 迁移过程中部分页已对齐、部分未对齐时，清单状态必须诚实；全部完成前不得宣称本功能存量迁移验收通过
- Logo 缺失或路径错误时，不得用随意装饰图替代品牌资产充数（修复资产引用，而非发明新视觉）
- 小屏设备上主数字与主按钮仍须可读可点，免责声明可弱化但仍可发现

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 系统 MUST 提供一套小程序计算页视觉契约，强制对齐品牌文档中的 Ink、Orange、Snow 三色，MUST NOT 另起主色板
- **FR-002**: Ink MUST 用于主文字、主操作按钮（如「计算」）实心填充及需要高对比的强调描边/结构；主按钮文字 MUST 为浅色（如 Snow/近白）以保证对比。Orange MUST 仅作有限强调（选中态、小点缀等），MUST NOT 作为主操作按钮的实心填充色，MUST NOT 将结果主数字整串着为 Orange。Snow MUST 作为页面背景/浅底体系基础
- **FR-003**: Orange MUST NOT 被用作大面积铺底以至于刺眼或压过主内容层级
- **FR-004**: 系统 MUST 将品牌色落地为统一、可命名的样式 token，供各计算页共享消费
- **FR-005**: 系统 MUST 提供单一共享样式入口，计算页与饮食路径入口/聚合页通过该入口获得通用版式能力（页边距、分区标题、表单字段、主按钮、次要选择态、错误提示、结果区、风险提示、免责声明）。选择态（性别/模式等）MUST 为：选中 = Orange 描边 + 浅底 + Ink 文字；未选中 = 中性浅灰底
- **FR-006**: 结果展示 MUST 遵循信息层级：主数字最突出 → 次要指标 → 说明/温馨提示 → 底部免责声明（弱化但不消失）。主数字填色 MUST 为 Ink；单位/附属标签 MUST 使用次级灰（Ink 衍生）；Orange 若出现 MUST 仅为极小点缀，MUST NOT 将整串主数字着为 Orange
- **FR-007**: 免提示或强调态 MUST 可辨识，且 MUST NOT 在视觉上压过主结果数字
- **FR-008**: Logo 使用 MUST 采用小程序侧品牌资产约定路径中的 icon/lockup；计算页不强制每页展示大 Logo，但 MUST NOT 用无关装饰淹没品牌识别
- **FR-009**: 存量页面 MUST 列入迁移清单，至少包含：饮食 Tab 入口页、身体指数工具箱聚合页、工具箱内 BMI / 基础代谢 / 体脂估算 / 腰臀比 / 1RM、以及热量缺口计算页
- **FR-010**: 迁移完成定义 MUST 包含：引用同一品牌 token 与共享样式入口；主按钮/主入口按钮为 Ink 实心；表单字段（若有）、结果主数字层级与免责声明权重（计算页）一致；入口页品牌资产使用约定资源
- **FR-011**: 迁移 MUST NOT 改变既有业务行为：点击计算、同页结果、修改输入后结果失效、不预填、表单校验与免责声明语义等保持不变
- **FR-012**: 实现过程允许分批改页，但本功能交付完成前 MUST 使迁移清单内全部目标页达到完成定义；MUST 能区分已对齐与待对齐状态，全部完成前 MUST NOT 宣称存量迁移验收通过
- **FR-013**: 此后新建计算类页面 MUST 遵循本视觉契约与品牌 token，MUST NOT 另起主色板
- **FR-014**: 本功能 MUST 提供简短的「新计算页如何引用品牌 token 与共享样式」说明，供后续功能直接遵循
- **FR-015**: 视觉气质 MUST 保持克制、新手友好、可信；MUST NOT 采用霓虹炫技、过度多层阴影卡片堆叠或医美风作为主视觉语言

### Key Entities

- **品牌色 Token**：Ink、Orange、Snow 及其用途边界（主色 / 强调 / 背景体系）
- **共享样式契约**：计算页通用结构与状态（字段、Ink 主按钮、Orange 描边选中态、错误、结果、风险、免责）
- **信息层级规则**：主数字 → 次要指标 → 说明 → 免责声明的固定优先级
- **迁移清单项**：单个存量页（含入口/聚合/计算页）的对齐状态与完成定义核对点
- **品牌 Logo 资产引用**：icon / 横向 lockup / 堆叠 lockup 的使用场景约定

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 抽查已对齐计算页，100% 主色仅来自 Ink / Orange / Snow（及衍生深浅/透明度），无第二套主色
- **SC-002**: 在工具箱任一样板页与热量缺口页之间对比，主按钮（Ink 实心）、表单字段、选中态（Orange 描边浅底）、结果主数字层级、免责声明权重观感一致（评审清单勾选）
- **SC-003**: 新手受试在无讲解情况下，于结果页 3 秒内能指出「最重要的数字」（该数字为 Ink 而非整串橙色）；免责声明在完整一屏或短滑后可见
- **SC-004**: 已对齐页主操作按钮为 Ink 实心；Orange 使用面积目视不超过装饰/强调角色（评审：无橙实心主按钮、无整屏大面积橙底喧宾夺主）
- **SC-005**: 本功能交付时，迁移清单内全部目标页（饮食 Tab 入口、工具箱聚合页、工具箱五计算页、热量缺口）均已达完成定义；未完成页不得计入完成数，亦不得关闭存量迁移验收
- **SC-006**: 迁移后抽样回归：每页至少一次合法计算路径与一次非法输入提示仍可用，业务行为无回归
- **SC-007**: 在主流手机宽度（约 375–430 逻辑像素）下，主路径无阻碍完成计算的布局问题
- **SC-008**: 后续新计算页需求文档可直接引用本契约；抽查说明文档含「必须用品牌三色 + 共享样式入口 + 信息层级」三点

## Assumptions

- 品牌三色与 Logo 文件以 `docs/brand/README.md` 为权威来源；本功能只消费、不扩展色板、不重做 Logo
- 小程序运行时 Logo 位于既有小程序品牌资源目录；与文档中的 logo 导出同源
- 表单/结果块可使用近白表面以提升可读性，但仍视为 Snow 体系内的表面色，不构成第四主色
- 由 Ink/Orange/Snow 衍生的灰阶与透明度（如分割线、次要文字）允许使用
- 迁移默认覆盖饮食路径：饮食 Tab 入口、工具箱聚合页、工具箱五计算页、热量缺口页；训练 Tab 等其他页面不在本功能必须清单内，可后续迭代
- 分批实现顺序建议：先共享 token/样式入口 + 一样板计算页，再入口/聚合页，再热量缺口与其余工具页；但本功能交付范围 MUST 覆盖清单全部页，不得把剩余页甩到未排期的 follow-up 才算「完成」
- 主操作按钮（「计算」等）与饮食路径主入口按钮统一为 Ink 实心 + 浅色字；Orange 不用于主按钮实心填充
- 选择态（性别/模式等）：选中为 Orange 描边 + 浅底 + Ink 字；未选中为中性浅灰底
- 结果主数字为 Ink，单位/标签为次级灰；Orange 不用于整串主数字着色（仅可选极小点缀）
- 既有「点击计算 / 同页结果 / 不预填」等交互不在本功能重新设计
- 本功能交付范围 MUST 覆盖迁移清单全部页（含饮食 Tab 与工具箱聚合页），不得以仅样板页宣称完成
- Web 官网视觉体系不在范围；Web 已有品牌资源目录，另开需求
- 不引入暗黑模式、插画体系、复杂动效系统

## Constitution Alignment _(mandatory)_

### In Scope / Out of Scope

- **In scope**: 微信小程序计算页视觉契约；品牌三色 token 化与共享样式入口；信息层级与通用计算页版式；饮食路径存量页（饮食 Tab 入口、工具箱聚合页、工具箱五计算页、热量缺口）迁移；增量遵循说明
- **Out of scope**: 修改 shared 计算逻辑；重做/扩展品牌色或 Logo 设计；Web 完整视觉体系；暗黑模式/插画/复杂动效；Taro/uni-app/重型 UI 库；社区/运营后台

### Stack & Shared Logic

- **Language**: TypeScript（页面逻辑保持既有 TS；本功能以样式与结构约定为主，无充分理由不新增纯 JS 业务源码）
- **Target package(s)**: `packages/miniprogram`（样式、页面引用、品牌资产使用）；只读对齐 `docs/brand`
- **Shared extraction**: 计算算法仍只在 `packages/shared`；本功能 MUST NOT 把视觉样式塞进 shared
- **Backend usage**: 无；纯前端视觉与静态品牌资产

### Compliance Checklist

- [x] No medical/treatment claims; disclaimer required on calculators/guidance（免责声明弱化但不消失）
- [x] Privacy: minimum necessary data only（本功能不新增数据采集）
- [x] No extreme diet / overload training recommendations（不借视觉鼓励极端行为）
- [x] Platform rules: WeChat category/content alignment (if miniprogram)
- [x] Brand: UI MUST consume docs/brand tokens; no alternate primary palette
