# Feature Specification: 热量缺口计算页面

**Feature Branch**: `003-calorie-plan-page`

**Created**: 2026-07-23

**Status**: Draft

**Input**: User description: "功能名称：热量缺口计算页面。所属模块：packages/miniprogram。完整的饮食热量方案计算功能，用户输入身体信息与运动习惯后，输出包含总热量、宏量营养素、三餐建议在内的可落地饮食方案。计算内核与类型定义复用 packages/shared，页面逻辑使用 TypeScript 编写。输入：性别、年龄、身高、体重；选填体脂率；日常活动水平、每周训练频率；训练目标减脂（温和/快速）、增肌、维持。输出：BMR、TDEE、目标摄入；宏量营养素（训练日/休息日碳水）；三餐分配与搭配示例；温馨提示。验收：计算逻辑完全复用 shared；减脂两档与风险提示；碳水区分训练日/休息日；表单校验完整；结果层级清晰；底部免责声明。"

## Clarifications

### Session 2026-07-23

- Q: 温馨提示是否按训练目标分化？ → A: 按目标分化：减脂显示减重速度+平台期+BMR 底线；增肌/维持显示各自适配提示（仍含 BMR 底线类合规提示，若适用）
- Q: 快速减脂的风险提示在何时展示？ → A: 仅结果区展示：选择快速减脂并计算后，结果页出现强化风险/注意事项（选择时不弹确认框）
- Q: 三餐热量分配比例如何定稿？ → A: 固定早:午:晚 = 3:4:3，本版本不可由用户改比例
- Q: 选填体脂率在结果中如何呈现？ → A: 仅当用户填写合法体脂时，结果摘要展示自报体脂，并注明未参与本次核心热量计算；未填写则不展示
- Q: 训练日/休息日碳水如何展示？ → A: 训练日与休息日各以推荐克数为主数字，其下附带参考区间

## User Scenarios & Testing _(mandatory)_

### User Story 1 - 填写身体与运动信息并生成饮食方案 (Priority: P1)

健身新手希望根据自身情况得到可执行的每日热量与营养建议。用户从饮食 Tab 进入「热量缺口计算」页，填写性别、年龄、身高、体重、日常活动水平、每周训练频率与训练目标（可选手填体脂率），点击计算后，在同页看到以目标摄入热量为核心的完整方案摘要。

**Why this priority**: 输入→计算→核心结果是本功能的主价值闭环，没有它则宏量、三餐与提示均无意义。

**Independent Test**: 仅实现表单、校验与核心热量结果（BMR、TDEE、目标摄入）即可演示完整主路径；宏量/三餐可后续叠加。

**Acceptance Scenarios**:

1. **Given** 用户位于饮食 Tab，**When** 进入热量缺口计算页，**Then** 看到空白表单，包含基础信息、运动信息与训练目标分区，体脂率为选填
2. **Given** 用户填写全部必填项且数值合法，**When** 点击「计算」，**Then** 同页展示 BMR、TDEE 与目标每日摄入总热量，且目标热量在信息层级上最突出
3. **Given** 结果已展示后用户修改任一输入，**When** 尚未再次点击「计算」，**Then** 原结果不再作为有效结果展示
4. **Given** 用户未登录，**When** 完成一次合法计算，**Then** 仍可正常获得结果（不强制登录）
5. **Given** 用户填写了合法体脂率并完成计算，**When** 查看结果摘要，**Then** 可见「用户自报体脂率」及未参与本次核心热量计算的说明
6. **Given** 用户未填体脂率并完成计算，**When** 查看结果摘要，**Then** 不展示体脂相关行

---

### User Story 2 - 按目标查看宏量营养素（含训练日/休息日碳水） (Priority: P1)

用户在获得目标热量后，需要知道每天吃多少蛋白质、碳水与脂肪，并理解训练日与休息日碳水应有所不同。结果区展示每日蛋白质、脂肪推荐克数，以及按训练日/休息日区分的碳水参考；页面结合用户填写的每周训练频率，帮助理解「一周内哪些天参考训练日碳水」。

**Why this priority**: 宏量拆解是「可落地饮食方案」的核心，且验收明确要求训练日/休息日碳水区分。

**Independent Test**: 在合法输入下计算，验证蛋白质、脂肪、训练日碳水、休息日碳水均展示，且与训练频率相关的说明可读。

**Acceptance Scenarios**:

1. **Given** 用户已成功生成方案，**When** 查看宏量营养素区，**Then** 可见每日蛋白质、脂肪推荐克数；训练日碳水与休息日碳水各自以推荐克数为主数字，并在其下展示参考区间，二者可区分、可同时看清
2. **Given** 用户填写了每周训练频率，**When** 查看碳水说明，**Then** 文案能帮助用户将「训练日/休息日」与自己的训练频率对应理解（例如一周训练 N 天时如何参考）
3. **Given** 用户选择不同训练目标后重新计算，**When** 对比宏量结果，**Then** 推荐数值随目标变化，且不出现互相矛盾的单一「万能碳水」表述

---

### User Story 3 - 减脂两档目标与风险提示 (Priority: P1)

减脂用户可在「温和减脂」与「快速减脂」两档间选择。选择快速减脂时，结果区明确展示更大热量缺口对应的风险/注意事项；无论哪一档，当目标摄入触及或被调整至不低于基础代谢时，用户能看到清晰提示，且不得鼓励低于基础代谢的极端摄入。

**Why this priority**: 产品边界与合规要求强制区分温和/快速缺口并标注风险；错误引导会造成健康与合规风险。

**Independent Test**: 分别选择温和减脂与快速减脂完成计算，验证两档结果差异可见，且快速档有额外风险提示；验证「不低于 BMR」提示在需要时出现。

**Acceptance Scenarios**:

1. **Given** 用户选择「温和减脂」并合法计算，**When** 查看结果，**Then** 目标摄入相对 TDEE 呈现温和缺口，并有对应说明
2. **Given** 用户选择「快速减脂」并合法计算，**When** 查看结果，**Then** 目标摄入相对温和档缺口更大，且结果页明确标注风险/注意提示（语气为劝阻极端而非鼓励激进）；选择该目标时 MUST NOT 弹出阻断式确认框
3. **Given** 用户仅切换目标至「快速减脂」但尚未点击计算，**When** 查看页面，**Then** 不因该选择单独弹出确认框，亦不提前展示完整结果区风险文案（以计算后结果为准）
4. **Given** 按目标推算的摄入将低于基础代谢，**When** 展示结果，**Then** 实际推荐摄入不低于 BMR，并展示「不建议低于基础代谢」类温馨提示
5. **Given** 用户选择增肌或维持体重，**When** 查看结果，**Then** 分别体现热量盈余或维持说明，且不套用减脂风险文案

---

### User Story 4 - 三餐建议与温馨提示 (Priority: P2)

用户希望把每日总热量落到三餐。结果区给出早/午/晚热量分配比例及简易食物搭配示例；同时按训练目标提供分化的温馨提示（减脂：合理减重速度、平台期、不低于 BMR；增肌/维持：各自适配提示，并在适用时保留不低于 BMR 类合规提示），降低新手执行焦虑。

**Why this priority**: 提升「可落地」体验，但可在核心热量与宏量之后交付。

**Independent Test**: 合法计算后确认三餐比例、搭配示例与（对应当前目标的）温馨提示区块均存在且可读；切换目标后提示内容随之匹配。

**Acceptance Scenarios**:

1. **Given** 方案已生成，**When** 查看饮食建议区，**Then** 可见三餐热量按早:午:晚 = 3:4:3 分配，且各餐热量可与目标总热量对应理解；用户无法在本页改该比例
2. **Given** 方案已生成，**When** 查看搭配示例，**Then** 可见简易、非极端的餐食搭配示例（非处方式强制菜单）
3. **Given** 用户选择温和或快速减脂并生成方案，**When** 查看温馨提示，**Then** 至少覆盖：合理减重速度参考、平台期应对思路、不建议低于基础代谢
4. **Given** 用户选择增肌或维持体重并生成方案，**When** 查看温馨提示，**Then** 展示与该目标适配的提示（如渐进增重或体重稳定相关），且 MUST NOT 展示减脂专用的「每周减重速度」表述；若触发 BMR 托底则仍可见不低于 BMR 类提示

---

### User Story 5 - 表单校验、结果可读性与免责声明 (Priority: P2)

用户填写错误或漏填时获得友好中文提示且不会看到错误结果；结果页信息层级清晰（目标热量最突出，其次 BMR/TDEE 与宏量，再次三餐与提示）；页面底部始终可见统一免责声明。

**Why this priority**: 保障新手可用性与合规底线，与主计算闭环互补。

**Independent Test**: 用非法/空输入验证校验；用合法结果检查视觉层级与底部免责声明。

**Acceptance Scenarios**:

1. **Given** 必填项为空、为负、非数字或明显不合理，**When** 点击「计算」，**Then** 展示友好中文校验提示，且不展示计算结果
2. **Given** 体脂率为空，**When** 其余必填合法并计算，**Then** 仍可成功出结果（体脂不阻塞计算）
3. **Given** 体脂率填写了非法值，**When** 点击「计算」，**Then** 给出该字段友好提示且不展示结果
4. **Given** 结果已展示，**When** 浏览结果区，**Then** 目标每日摄入总热量在视觉上优先于次要说明；新手无需滚动到页尾才能看到核心数字
5. **Given** 任意结果已展示，**When** 查看页面底部，**Then** 可见统一免责声明（语义等价于「仅供参考，不构成专业健身指导，请根据自身身体状况量力而行」）

---

### Edge Cases

- 年龄、身高、体重为 0、负数、非数字或极端超大值时，阻止计算并提示
- 活动水平或训练目标未选择时，阻止计算并提示
- 每周训练频率为 0（完全不训练）时，仍可计算；碳水区应能理解「可全部参考休息日」或等价友好说明
- 每周训练频率填满 7 天时，碳水区应能理解「多数天可参考训练日」或等价友好说明
- 选填体脂率留空与填写合法值均可计算；填写非法体脂率则拦截
- 目标推算低于 BMR 时，展示托底后的目标热量与明确提示，不展示鼓励更低摄入的文案
- 快速减脂场景下，提示侧重风险与可持续性，不得宣传「越饿越好」或极端节食
- 增肌/维持目标下，温馨提示不得出现减脂专用的每周减重速度文案
- 宏量结构过紧（蛋白质+脂肪已接近或超过目标热量）时，仍给出可理解结果说明，不出现负数克数展示
- 用户修改输入后未重算时，不得继续把旧方案当作有效结果

## Requirements _(mandatory)_

### Functional Requirements

- **FR-001**: 系统 MUST 在微信小程序饮食相关入口提供「热量缺口计算」页面，供用户完成输入与查看饮食热量方案
- **FR-002**: 表单 MUST 收集必填项：性别、年龄、身高、体重、日常活动水平、每周训练频率、训练目标；MUST 支持选填体脂率。当用户填写合法体脂率时，结果摘要 MUST 展示该自报值并注明未参与本次核心热量（BMR/TDEE/目标摄入）计算；未填写时 MUST NOT 展示体脂行
- **FR-003**: 训练目标 MUST 至少包含：温和减脂、快速减脂、增肌、维持体重；减脂 MUST 明确分为温和与快速两档
- **FR-004**: 用户触发计算后，系统 MUST 展示核心数据：基础代谢（BMR）、每日总消耗（TDEE）、目标每日摄入总热量
- **FR-005**: 系统 MUST 展示宏量营养素建议：每日蛋白质、脂肪推荐克数；训练日碳水与休息日碳水 MUST 可区分展示，且各自以推荐克数为主数字，其下附带参考区间
- **FR-006**: 系统 MUST 结合用户填写的每周训练频率，提供帮助用户理解训练日/休息日碳水如何对应到自己一周安排的说明
- **FR-007**: 系统 MUST 展示饮食建议：三餐热量按固定比例早:午:晚 = 3:4:3 分配（并展示对应热量），以及简易搭配示例；本版本 MUST NOT 提供用户自定义或切换三餐比例
- **FR-008**: 系统 MUST 按训练目标分化展示温馨提示：温和/快速减脂 MUST 至少包含每周合理减重速度参考、平台期应对思路、不建议低于基础代谢；增肌与维持 MUST 展示各自适配提示，MUST NOT 使用减脂专用减重速度文案；任一目标在触发 BMR 托底时 MUST 展示不低于基础代谢类提示
- **FR-009**: 选择快速减脂并完成计算后，系统 MUST 在结果区额外展示相对温和减脂更明确的风险/注意事项提示；温和减脂亦须有基础说明，但语气强度低于快速档。切换至快速减脂时 MUST NOT 使用阻断式确认弹窗；风险强化提示以计算结果展示为时机
- **FR-010**: 当按目标推算的摄入低于基础代谢时，系统 MUST 将推荐摄入调整为不低于 BMR，并明确提示用户不建议低于基础代谢
- **FR-011**: 所有热量与宏量计算逻辑 MUST 完全复用共享计算内核，页面 MUST NOT 重复实现 BMR、TDEE、目标摄入、宏量拆解等算法
- **FR-012**: 页面层 MUST 复用共享包已有入参与返回值类型定义，MUST NOT 重复声明与共享内核等价的业务类型；表单状态与计算结果 MUST 使用 TypeScript 类型标注
- **FR-013**: 表单 MUST 做完整校验：空值、负数、非数字、未选择枚举项、非法体脂率等给出友好中文提示，且不输出计算结果
- **FR-014**: 结果展示 MUST 信息层级清晰：目标每日摄入总热量为重点突出项；BMR/TDEE 与宏量次之；三餐建议与温馨提示再次之，便于新手扫读
- **FR-015**: 每次进入页面 MUST 为空白表单（或不依赖历史输入的初始态）；本版本 MUST NOT 预填或自动恢复上次输入
- **FR-016**: 采用「点击计算 → 同页展示结果」交互；修改任一相关输入后，原结果 MUST 失效直至再次计算
- **FR-017**: 页面底部 MUST 统一展示免责声明，语义等价于产品合规文案：「仅供参考，不构成专业健身指导，请根据自身身体状况量力而行」
- **FR-018**: 解读、建议与提示文案 MUST 符合新手友好原则，MUST NOT 包含医疗治疗宣称，MUST NOT 推荐极端节食或有害健康做法
- **FR-019**: 页面 MUST 适配主流手机屏幕，核心路径无明显布局错乱或横向滚动障碍
- **FR-020**: 本功能 MUST NOT 强制登录，MUST NOT 将本次计算的身体数据上传云端作为计算前置条件（本地完成计算即可）

### Key Entities

- **身体基础信息**：性别、年龄、身高、体重；可选体脂率（用户自报，选填；有值时仅结果摘要展示，不参与本版核心热量计算）
- **运动习惯**：日常活动水平、每周训练频率
- **训练目标**：温和减脂 / 快速减脂 / 增肌 / 维持体重
- **核心热量结果**：BMR、TDEE、目标每日摄入总热量；是否触发 BMR 托底及对应提示
- **宏量营养素方案**：蛋白质、脂肪；休息日/训练日碳水（推荐克数为主 + 参考区间）及面向用户的说明
- **三餐建议**：早/午/晚按 3:4:3 固定比例分配的热量与简易搭配示例
- **温馨提示集**：按目标分化的科普向提示（减脂：减重速度、平台期、不低于 BMR；增肌/维持：目标适配提示 + 适用时的 BMR 底线提示）
- **表单校验反馈**：字段级或表单级错误提示
- **免责声明**：结果页底部统一合规文案

## Success Criteria _(mandatory)_

### Measurable Outcomes

- **SC-001**: 用户从饮食相关入口进入后，可在 2 分钟内完成一次合法填写并看到完整核心方案（目标热量 + 宏量）
- **SC-002**: 首次使用的新手在无外部说明的情况下，能识别「目标每日摄入」为最重要数字，并理解训练日与休息日碳水不同（各自可见主推荐克数及下方参考区间）
- **SC-003**: 使用相同合法输入多次计算，页面展示的 BMR、TDEE、目标热量与宏量数值与共享计算内核结果完全一致（无页面自研算法偏差）
- **SC-004**: 温和减脂与快速减脂在相同身体与活动输入下，目标摄入可区分；快速档在计算后的结果页 100% 可见风险/注意提示，且选择该目标时无阻断式确认弹窗
- **SC-005**: 对空值、负数、非法输入及未选择必填枚举的校验用例，100% 阻止计算并给出可理解中文提示
- **SC-006**: 体脂率留空时计算成功且结果中无体脂行；填写合法体脂时计算成功且结果摘要展示自报体脂及「未参与本次核心热量计算」说明；非法体脂 100% 被拦截
- **SC-007**: 当推算摄入低于 BMR 的场景被触发时，展示的目标摄入不低于 BMR，且用户可见对应提示
- **SC-008**: 合法结果页均可见按 3:4:3 分配的三餐建议、与当前目标匹配的温馨提示与底部统一免责声明；减脂结果含减重速度类提示，增肌/维持结果不含减脂专用减重速度表述
- **SC-009**: 在主流手机宽度（约 375–430 CSS 像素）下，主路径可读可操作，无阻碍完成计算的布局问题
- **SC-010**: 抽样检查结果与提示文案：无医疗治疗宣称、无极端节食鼓励表述

## Assumptions

- 入口归属饮食 Tab 能力范围；具体菜单位置可与现有饮食首页/工具入口对齐，本版本不强制独立 Tab
- 交互采用「点击计算 → 同页展示结果」，与身体指数工具箱一致，降低学习成本
- 计量单位采用国内常见默认：身高 cm、体重 kg、体脂率 %、热量 kcal；本版本不提供单位切换
- 性别选项至少覆盖男/女，以匹配共享内核入参
- 日常活动水平选项与共享内核活动分档一致（久坐至非常活跃等通俗文案映射）
- 每周训练频率以「每周训练天数」（0–7）理解；用于解释训练日/休息日碳水如何落地，不另造第二套宏量算法
- 训练日/休息日碳水展示形态：各以推荐克数为主数字，其下附带可读参考区间；区间须来自共享内核已有范围能力或由其结果等价派生，页面不得自研第二套宏量算法
- 核心 BMR 按共享内核既有方式计算（产品边界所述 Mifflin-St Jeor 路径）；选填体脂率不参与本版核心热量计算；仅当用户填写合法值时在结果摘要展示自报体脂并注明未参与计算；未填写则不展示。若共享内核后续支持体脂参与计算，以内核为准且页面仍不得本地重写
- 减脂温和/快速缺口、增肌盈余、维持等幅度遵循共享内核已定义的目标热量调整策略（与产品边界中的温和 300–500、快速 500–700、增肌 200–400 大卡区间一致）
- 三餐热量分配固定为早:午:晚 = 3:4:3，本版本不提供比例切换；搭配示例为科普向示意，非个性化处方；示例随总热量按该比例展示各餐热量即可
- 合理减重速度温馨提示仅用于减脂目标，采用科普常用量级（例如每周约 0.5–1% 体重或等价通俗表述），并强调个体差异与量力而行；增肌/维持使用各自适配的温馨提示文案
- 快速减脂的强化风险提示仅在计算结果区展示；选择目标时不弹阻断确认
- 本版本不预填、不记住上次输入；与「我的-身体数据记录」联动不在范围
- 共享计算内核（BMR、TDEE、目标摄入、宏量计划）及其类型由 `packages/shared` 提供；若缺口须先在 shared 按 TDD 补齐后再供页面调用
- Web 端完整热量方案页复刻不在本功能范围（Web 仅按产品边界开放极简热量测算）
