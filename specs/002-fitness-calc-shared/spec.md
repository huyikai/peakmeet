# Feature Specification: 健身核心计算共享库

**Feature Branch**: `002-fitness-calc-shared`

**Created**: 2026-07-23

**Status**: Draft

**Input**: User description: "使用 TypeScript 实现全项目统一的健身计算内核，所有算法均为纯函数，无任何平台依赖，供小程序端、Web 端共同复用，保证两端计算结果与类型定义 100% 一致。全程严格遵循 TDD，单元测试全覆盖。包含 BMI、BMR、TDEE、目标摄入、宏量营养素、1RM、体脂率估算、腰臀比等能力。"

## Clarifications

### Session 2026-07-23

- Q: 体脂率估算公式（输入含腰围、体重、性别、年龄）？ → A: 采用项目固定的四输入估算式（腰围+体重+性别+年龄），系数与样例在 plan 写死
- Q: 非法入参时的失败表达方式？ → A: 返回成功/失败联合结果（含错误码与友好文案），不抛异常
- Q: Epley 1RM 的合法次数上限？ → A: 次数须为 1–12（含）；超出则失败
- Q: 热量与宏量克数的对外取整规则？ → A: 热量整数 kcal；蛋白/脂肪/碳水克数保留 1 位小数
- Q: 区间类参数是否允许调用方覆盖代表值？ → A: 仅用固定中点代表值；调用方不可覆盖（结果可返回区间上下限供 UI 说明）

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 身体指数与基础代谢可一致计算 (Priority: P1)

作为 PeakMeet 用户（经小程序或官网工具），我希望根据身高体重等基础信息得到 BMI 判定与通俗解读，并得到按性别区分的基础代谢（BMR）与结合活动水平的每日总消耗（TDEE），以便后续热量目标有共同起点。

**Why this priority**: BMI/BMR/TDEE 是饮食计算与工具箱的底座；无此无法做目标热量与宏量分配。

**Independent Test**: 仅交付本故事对应计算能力时，给定同一组合法输入，BMI 分类标签、BMR、TDEE 结果可独立验证且重复调用结果相同。

**Acceptance Scenarios**:

1. **Given** 合法的身高（cm）与体重（kg），**When** 计算 BMI，**Then** 返回数值、标准范围判定标签与通俗解读文案。
2. **Given** 合法的性别、年龄、身高、体重，**When** 按 Mifflin-St Jeor 规则计算 BMR，**Then** 男女公式区分正确，结果为正数热量。
3. **Given** 已知 BMR 与五档活动水平之一（久坐/轻度/中度/高度/极高），**When** 计算 TDEE，**Then** 返回与活动档位对应的每日总消耗。
4. **Given** 同一输入重复计算两次，**When** 比较结果，**Then** 数值与标签完全一致（纯函数、无副作用）。

---

### User Story 2 - 目标摄入热量与「不低于 BMR」兜底 (Priority: P1)

作为用户，我希望按减脂温和、减脂快速、增肌或维持四类目标得到每日建议摄入热量；若减脂后低于基础代谢，系统自动兜底到不低于 BMR 并给出明确提示，避免不安全建议。

**Why this priority**: Constitution 强制「摄入不得低于基础代谢」；是合规与产品安全红线。

**Independent Test**: 在已知 TDEE/BMR 下，四类目标均可算出建议摄入；构造会低于 BMR 的减脂场景时必触发兜底与提示。

**Acceptance Scenarios**:

1. **Given** 有效 TDEE 与目标「维持」，**When** 计算目标摄入，**Then** 建议摄入等于 TDEE（对外为整数大卡）。
2. **Given** 有效 TDEE 与「减脂温和」，**When** 计算，**Then** 缺口等于相对 TDEE 减少 300–500 大卡区间的固定中点（调用方不可覆盖）。
3. **Given** 有效 TDEE 与「减脂快速」，**When** 计算，**Then** 缺口等于减少 500–700 大卡区间的固定中点。
4. **Given** 有效 TDEE 与「增肌」，**When** 计算，**Then** 盈余等于增加 200–400 大卡区间的固定中点。
5. **Given** 减脂计算结果将低于 BMR，**When** 计算完成，**Then** 最终建议摄入被提升至不低于 BMR，并返回可展示的提示说明。

---

### User Story 3 - 宏量营养素按目标与训练日/休息日分配 (Priority: P2)

作为用户，我希望在已知目标摄入与体重后，得到蛋白质、脂肪、碳水的克数建议，且碳水区分训练日与休息日，以便安排饮食结构。

**Why this priority**: 依赖目标热量；是饮食 Tab 核心输出，但可在热量能力之后独立验收。

**Independent Test**: 固定热量、体重与目标类型，可验证蛋白/脂肪落在约定 g/kg 区间、碳水由剩余热量折算，且训练日碳水高于休息日。

**Acceptance Scenarios**:

1. **Given** 减脂目标与体重，**When** 分配宏量，**Then** 蛋白质按 1.8–2.2 g/kg、脂肪按 0.8–1.0 g/kg 落在约定代表值规则内。
2. **Given** 增肌目标与体重，**When** 分配宏量，**Then** 蛋白质按 2.0–2.4 g/kg、脂肪按 0.8–1.0 g/kg 落在约定代表值规则内。
3. **Given** 同一热量与宏量设定，**When** 分别按训练日与休息日计算碳水，**Then** 训练日碳水克数相对休息日上浮 10%–20%（按约定代表值）。
4. **Given** 蛋白与脂肪热量已占部分总热量，**When** 计算碳水，**Then** 碳水由剩余热量按 4 kcal/g 折算，不为负。

---

### User Story 4 - 力量与围度相关指标 (Priority: P2)

作为用户，我希望用常见训练数据估算 1RM，并用腰围、体重、性别、年龄估算体脂率，以及用腰围臀围计算腰臀比与健康风险等级，用于工具箱解读。

**Why this priority**: 身体指数工具箱能力；与热量链路相对独立，可并行交付。

**Independent Test**: 各指标可单独用夹具输入验证数值与风险/解读标签。

**Acceptance Scenarios**:

1. **Given** 合法的一次接近力竭重量与次数（次数 1–12），**When** 按 Epley 规则估算 1RM，**Then** 返回正数估算值。
2. **Given** 合法的腰围、体重、性别、年龄，**When** 用项目固定四输入估算式计算体脂率，**Then** 返回百分比数值与约定解读标签，且与测试夹具期望一致。
3. **Given** 合法的腰围与臀围，**When** 计算腰臀比，**Then** 返回比值与健康风险等级标签（按性别区分阈值，见 Assumptions）。

---

### User Story 5 - 非法输入有明确失败信息 (Priority: P1)

作为接入方（小程序/Web），我希望对空值、负数、缺失必填项等非法输入得到可预测的失败结果与友好说明，而不是静默错误数值。

**Why this priority**: 防止错误热量建议；与全部计算共用。

**Independent Test**: 对每个公开计算能力至少覆盖一类非法入参，断言失败语义清晰、无部分脏写。

**Acceptance Scenarios**:

1. **Given** 身高或体重为 0/负数/缺失，**When** 调用 BMI 或 BMR 相关计算，**Then** 返回失败联合结果（含错误码与友好文案），不抛异常，且不产出看似有效的热量建议。
2. **Given** 活动水平或目标类型为未支持枚举值，**When** 计算 TDEE 或目标摄入，**Then** 返回失败联合结果（含错误码与友好文案），不抛异常。
3. **Given** 1RM 次数 ≤0 或 >12（或非整数），**When** 估算，**Then** 返回失败联合结果拒绝计算，不抛异常。

---

### Edge Cases

- 减脂快速缺口后低于 BMR：必须兜底到 ≥ BMR，并带提示（不得静默给出低于 BMR 的摄入）。
- 极低体重/极高体重、未成年年龄：计算可进行但解读标签 MUST 提示「仅供参考、非医疗建议」；不提供治疗/康复表述。
- 宏量分配后剩余热量不足以支撑碳水：碳水为 0 或按规则的最小非负值，并可选提示热量结构紧张。
- 训练日/休息日切换仅影响碳水，不改变蛋白与脂肪的 g/kg 规则（除非后续规格修订）。
- 所有成功结果的解读文案 MUST 可附带统一免责声明语义：「仅供参考，不构成专业健身指导…」（由调用方展示或由返回结构提供标准文案常量）。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 共享计算内核 MUST 提供 BMI 计算，包含数值、标准范围判定与通俗解读标签。
- **FR-002**: MUST 提供 BMR 计算，采用 Mifflin-St Jeor 公式并区分男性/女性。
- **FR-003**: MUST 提供 TDEE 计算，活动水平枚举恰好五档：久坐、轻度、中度、高度、极高，各档对应固定活动系数（系数表见 Assumptions）。
- **FR-004**: MUST 提供目标摄入计算，支持四类目标：减脂温和、减脂快速、增肌、维持；缺口/盈余幅度遵循 Constitution 区间；对外建议摄入为整数 kcal。
- **FR-005**: 目标摄入结果 MUST 强制保证最终建议摄入 ≥ BMR；若调整发生，MUST 返回可展示提示。
- **FR-006**: MUST 提供宏量营养素分配：蛋白与脂肪按目标类型的 g/kg 区间规则；碳水由剩余热量折算，并区分训练日与休息日（训练日碳水相对休息日上浮 10%–20%）；对外克数 MUST 保留 1 位小数。
- **FR-007**: MUST 提供基于 Epley 规则的 1RM 估算；合法次数范围为 1–12（含），超出该范围 MUST 返回失败联合结果。
- **FR-008**: MUST 提供体脂率估算；输入 MUST 包含腰围、体重、性别、年龄（四项均参与计算）；采用项目固定估算式（非临时改用仅部分输入的公开公式），系数与基准样例在实现计划中写死；输出百分比与解读标签。
- **FR-009**: MUST 提供腰臀比计算与按性别区分的健康风险等级判定。
- **FR-010**: 全部计算能力 MUST 为纯函数：相同输入→相同输出；无 I/O、无全局可变状态、无平台 API。
- **FR-011**: 非法入参 MUST 以成功/失败联合结果表达（含稳定错误码与面向展示的友好文案）；公开计算 API MUST NOT 抛异常；不得返回无标注的错误数值。
- **FR-012**: 成功结果所附解读/标签 MUST 避免医疗承诺用语；须可配合统一免责声明使用。
- **FR-013**: 所有对外入参、出参与枚举 MUST 有完整、稳定的类型契约，并通过统一入口导出，供多端引用且定义一致。
- **FR-014**: 计算内核 MUST 零第三方运行时依赖；构建产物 MUST 包含完整类型声明文件。
- **FR-015**: 每个公开计算能力 MUST 有单元测试，覆盖正常、边界与异常入参；共享计算相关测试覆盖率 MUST 达到 100%；开发 MUST 遵循先失败测试再实现再重构的顺序。

### Key Entities

- **BodyMetricsInput**: 性别、年龄、身高、体重等基础身体指标；可选体脂率。
- **ActivityLevel**: 五档活动水平枚举（久坐/轻度/中度/高度/极高）。
- **NutritionGoal**: 四类目标枚举（减脂温和/减脂快速/增肌/维持）。
- **BmiResult**: BMI 值、范围判定、解读标签。
- **EnergyResult**: BMR、TDEE、目标摄入、是否触发 BMR 兜底、提示文案。
- **MacroPlan**: 蛋白/脂肪/碳水克数；训练日与休息日碳水。
- **StrengthEstimate**: 1RM 估算值及输入重量、次数。
- **BodyCompEstimate**: 体脂率估算值与解读；腰臀比与风险等级。
- **CalcError**: 标准化失败分支（稳定错误码 + 面向展示的友好文案）；与成功结果组成联合返回类型。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 对同一套标准夹具输入，BMI、BMR、TDEE、目标摄入、宏量、1RM、体脂率、腰臀比在两次独立调用中结果 100% 一致。
- **SC-002**: 小程序与 Web 若均接入本内核，对同一输入展示的核心数值（热量、克数、比值）差异为 0（允许仅展示层取整方式按文档统一）。
- **SC-003**: 构造「减脂后低于 BMR」的夹具用例时，100% 触发兜底且提示可见；无用例输出低于 BMR 的最终建议摄入。
- **SC-004**: 每个公开计算能力至少各有正常、边界、异常三类自动化用例且全部通过；共享计算相关覆盖率报表为 100%。
- **SC-005**: 评审确认无平台 API、无第三方运行时依赖；类型声明随构建产物完整可用。
- **SC-006**: 抽检解读文案：无「治疗/康复/治病/治愈」等医疗表述；免责声明语义可被调用方获取或展示。

## Assumptions

- 本功能交付范围为共享计算内核与测试；不包含小程序/Web 页面 UI（调用方可后续接入）。
- 活动系数（示例默认，计划可微调但须锁定测试基准）：久坐 1.2、轻度 1.375、中度 1.55、高度 1.725、极高 1.9。
- 区间类参数（缺口、盈余、g/kg、训练日碳水上浮）**仅使用固定中点代表值**；调用方 **不可** 覆盖代表值；成功结果中可返回所用区间上下限，便于 UI 说明。覆盖能力留待后续规格。
- 热量取整：对外成功结果中的热量（BMR/TDEE/目标摄入等）为**整数 kcal**；宏量克数（蛋白/脂肪/碳水）**保留 1 位小数**；取整方式在 plan 写死（如四舍五入）并进入测试期望。
- 体脂率估算：采用**项目固定四输入估算式**（腰围 cm、体重 kg、性别、年龄均参与）；禁止在实现中擅自改为仅使用部分字段的其他公开公式。具体系数、单位换算与基准样例在 `/speckit-plan` 的 research 中写死并进入测试夹具。
- 腰臀比风险阈值：采用常用性别区分切点（如男性 ≥0.9、女性 ≥0.85 为较高风险档的起点），计划阶段写死分级表。
- Epley：`1RM = w × (1 + r/30)`（w 重量，r 次数）；合法次数 **1–12（含）**，≤0、>12 或非正整数次数均失败。
- 现有 `getPeakMeetPing` 冒烟导出保留，不删除；本功能新增计算模块与之并存。
- 单位：身高 cm、体重 kg、围度 cm、热量 kcal、力量单位与入参一致（默认 kg）。
- 错误模型：全部公开计算函数返回 `Result` 式联合类型（成功数据 | `CalcError`）；禁止 throw；错误码表在 plan/contracts 锁定。

## Constitution Alignment *(mandatory)*

### In Scope / Out of Scope

- **In scope**: `packages/shared` 内健身纯计算能力、类型契约、TDD 单测与构建类型声明；供小程序与 Web 复用。
- **Out of scope**: 小程序/Web UI、云函数热量计算、用户数据存储、三餐具体菜谱、医疗诊断、真实营养师方案。

### Stack & Shared Logic

- **Language**: TypeScript；完整 interface/type；禁止 `any`；枚举用 TS 枚举或字符串联合；统一类型导出入口
- **Target package(s)**: `packages/shared`（唯一实现地）
- **Shared extraction**: 本功能全部算法均在 shared；业务端禁止平行实现
- **Backend usage**: 无；纯计算 MUST NOT 走云函数

### Compliance Checklist

- [x] No medical/treatment claims; disclaimer required on calculators/guidance（解读 + 统一免责声明语义）
- [x] Privacy: minimum necessary data only（本功能不采集/不存储用户数据，仅处理调用方传入参数）
- [x] No extreme diet / overload training recommendations（快速减脂仍受 BMR 兜底约束；不鼓励低于 BMR）
- [x] Platform rules: N/A for pure shared lib（UI 接入时再对齐）
